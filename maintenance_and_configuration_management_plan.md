## 1. 프로젝트 개요

### 1.1 프로젝트명 및 목표

* **프로젝트명**: LoGym (Location-based Gym Community & Workout Logger)
* **목표**: 본 프로젝트는 기존 피트니스 센터의 기구 브랜드, 관리 상태 등 인프라 정보가 비대칭적으로 제공되는 한계를 해결하기 위해 기획되었다. 실제 헬스장의 인프라 정보(기구 맵)와 개인의 운동 기록(고정밀 타이머, 세트 데이터)을 유기적으로 결합하여, 사용자가 운동 장소를 탐색하는 비용을 줄이고 운동 세션을 안전하게 기록 및 분석할 수 있도록 지원한다. 더불어 소셜 피드를 통해 사용자 간의 지지를 이끌어내는 피트니스 소셜 플랫폼을 지향한다.
* **기술 스택**: React (Vite), Supabase (PostgreSQL, GoTrue Auth, Realtime, RLS), Kakao Map SDK, Tailwind CSS, PWA

## 2. 유지보수 계획

### 2.1 유지보수 방법

본 절은 IEEE Std 14764에서 정의한 소프트웨어 유지보수의 4대 분류 체계를 LoGym 프로젝트에 적용하여 수립한다. 서비스 배포 이후 발생하는 모든 유지보수 활동은 아래의 유형 중 하나로 분류하여 처리할 계획이다.

#### 2.1.1 수정 유지보수 (Corrective Maintenance)

배포 후 발견되는 소프트웨어 결함(Defect), 장애(Failure), 오류(Error)를 식별하고 수정하는 활동이다.

* **적용 시나리오**:
  * 스톱워치의 `lastStartTime` 파싱 오류로 타이머가 비정상 복원되는 버그 발생 시
  * Supabase RLS 정책의 예외로 인해 비인가 사용자의 데이터 Insert가 허용되는 보안 오류 발생 시
  * Kakao Map SDK 마커 렌더링이 특정 모바일 브라우저(구 Safari 등)에서 미작동하는 경우
* **대응 방침**: GitHub Issues에 `bug` 레이블로 등록하고, 긴급도 분류 기준에 따라 핫픽스(Hotfix) 브랜치 또는 정기 릴리즈 사이클에 포함하여 수정할 계획이다.

#### 2.1.2 적응 유지보수 (Adaptive Maintenance)

운영 환경, 외부 의존성, 플랫폼의 변화에 대응하여 소프트웨어를 수정하는 활동이다.

* **적용 시나리오**:
  * Supabase API 메이저 버전 업그레이드(예: JS SDK v2 → v3) 시 클라이언트 코드 전면 대응
  * Kakao Map SDK의 오버레이 API 정책 변경에 따른 마커·BottomSheet 인터랙션 로직 수정
  * iOS/Android 모바일 브라우저의 새로운 보안 정책(예: Location API 권한 모델 변경) 적용 시
  * PWA의 Service Worker API 표준 변경에 따른 캐싱 전략 수정
* **대응 방침**: 외부 의존 라이브러리(Supabase, Kakao Map SDK 등)의 공식 변경 로그(Changelog)를 주기적으로 모니터링하고, 영향도 분석 후 적응 패치를 계획적으로 반영할 계획이다.

#### 2.1.3 기능 보강 유지보수 (Perfective Maintenance)

서비스 운영 중 수집되는 사용자 피드백 및 요구사항을 바탕으로 기존 기능을 개선하거나 새로운 기능을 추가하는 활동이다.

* **적용 시나리오**:
  * 사용자 피드백을 통해 운동 세션 기록 화면에 '이전 세션 기록 불러오기' 기능 추가 요구 발생 시
  * 소셜 피드에 운동 영상 첨부 기능, 사용자 프로필 페이지 등의 신규 기능 요청 발생 시
  * 대시보드 시각화(히트맵) 의 색상 범위 및 정밀도 향상 개선 요구 발생 시
* **대응 방침**: 수집된 기능 개선 요구사항은 GitHub Issues에 `enhancement` 레이블로 등록한다. 우선순위 평가 후 스프린트 백로그에 반영하여 계획적으로 개발할 계획이다.

#### 2.1.4 예방 유지보수 (Preventive Maintenance)

현재 당장 오류는 아니지만, 장기적인 유지보수성·성능·보안 저하를 방지하기 위한 선제적 개선 활동이다.

* **적용 시나리오**:
  * `requestAnimationFrame` 루프의 메모리 누수(Memory Leak) 위험 요소를 사전에 진단하고 컴포넌트 언마운트 시 정리(cleanup) 로직 강화
  * 서비스 규모 확장에 따른 DB 쿼리 인덱스 최적화 및 RLS 정책 리뷰
  * 누적된 기술 부채(Technical Debt) 해소를 위한 정기 리팩토링 스프린트 운영
* **대응 방침**: 분기별 1회 이상 코드베이스 정적 분석(Static Analysis) 및 기술 부채 리뷰 세션을 운영할 계획이다. 시점 명시: 10주 개발 일정 중에는 스프린트 종료 시점 자체 선에 주기적 코드 리뷰 및 성능 관리를 수행하며, Vercel 프로덕션 배포 후에는 실제 분기(약 3개월) 단위로 예방 유지보수 세션을 수행할 계획이다.

### 2.2 유지보수 절차

유지보수 요건이 발생한 경우, 아래의 5단계 표준 절차를 준수하여 처리할 계획이다. 이 절차는 배포 이후 운영 단계에서 모든 유지보수 활동의 기준이 되는 워크플로우이다.

```
[STEP 1: 문제 인식 및 접수]
  사용자 버그 리포트 · 모니터링 알람 · 개발팀 코드 리뷰 중 발견
  → GitHub Issues에 [보고 양식 템플릿]을 활용하여 등록
        ↓
[STEP 2: 분석 및 분류]
  유지보수 유형 분류 (수정 / 적응 / 기능 보강 / 예방)
  긴급도 분류 (Critical → 즉시 핫픽스 / High/Medium/Low → 스프린트 편입)
  영향 범위(Component, DB, API) 분석
        ↓
[STEP 3: 수정 계획 수립 및 작업 착수]
  스프린트 백로그 편입 (중요 변경의 경우 자체 검토)
  변경 범위 문서화 (CR 이슈 등록)
  `main` 브랜치 또는 임시 `hotfix/*` 브랜치에서 로컬 작업 착수
        ↓
[STEP 4: 수정 구현 및 자체 검증]
  코드 수정 구현
  단위/통합 테스트 로컬 수행
  자체 체크리스트(Self-Review) 검증 완료
  필요 시 Vercel Preview 환경을 통한 회귀 테스트(Regression Test)
        ↓
[STEP 5: 릴리즈 및 사후 모니터링]
  `main` 브랜치 푸시 및 Vercel 프로덕션 파이프라인 자동 트리거
  GitHub Release에 변경 내역(Changelog) 기록
  배포 후 24시간 이상 모니터링 수행
  Issues 항목 닫기(Closed) 및 완료 처리
```

### 3.2 버전관리 전략

#### 3.2.1 브랜치 전략 (Trunk-based Development, 단일 브랜치 중심)

**[전략 채택 근거]**

프로젝트 개발 계획서(제안서)에 명시된 바와 같이, 본 프로젝트는 **1인 풀스택 개발자가 10주(2.5 M/M)의 한정된 일정 내에 애자일 스크럼 기반의 2주 단위 스프린트**로 진행되는 소규모 프로젝트이다. 이 특성을 고려하여 버전관리 전략을 아래와 같이 결정하였다.

일반적인 팀 개발 환경에서는 `main / develop / feature / release / hotfix`의 다중 브랜치 구조(Git-flow)나 `main / feature` 구조(GitHub Flow)를 적용하여 **팀원 간의 코드 충돌 방지와 리뷰 프로세스**를 보장한다. 그러나 1인 개발 환경에서 `develop` 브랜치로 PR을 열고 스스로 승인하는 방식은 **실질적인 품질 통제 효과 없이 행정적 오버헤드만 발생**시키므로 이를 배제한다.

따라서 커밋의 지속적인 변경 이력 추적이라는 형상관리의 본질적 목적을 유지하면서도 **개발 속도와 일정 준수**를 우선시하여, `main` 단일 브랜치 중심의 **Trunk-based Development** 방식을 채택할 계획이다. 단, 긴급 장애 발생 시에는 `hotfix/*` 브랜치를 한시적으로 활용한다.

```
main (유일한 통합 브랜치 · Production 배포 기준선)
  └── hotfix/critical-rls-bypass  (긴급 장애 발생 시에만 한시적으로 분기 후 즉시 병합)
```

* **`main` 브랜치**: 스프린트의 모든 개발 작업이 직접 커밋되는 단일 통합 브랜치이다. 각 스프린트 완료 시점에 GitHub Release 태그를 등록하여 배포 기준선(Baseline)을 명시적으로 관리할 계획이다.
* **`hotfix/*` 브랜치**: Critical(P0) 또는 High(P1) 등급의 긴급 장애 발생 시에만 `main`에서 한시적으로 분기하여 수정 후 즉시 `main`에 병합한다. 평시에는 사용하지 않는다.

> **[일반 팀 프로젝트와의 비교]** 2인 이상의 팀 환경으로 프로젝트가 확장될 경우, 코드 충돌 방지 및 코드 리뷰 프로세스 확보를 위해 `develop` 브랜치를 도입하고 `feature/*` 브랜치 기반의 GitHub Flow 또는 Git-flow로 전환하는 것을 권고한다.

#### 3.2.2 커밋 메시지 컨벤션 (Semantic Commit, 간소화 적용)

**[전략 채택 근거]**

1인 스프린트 환경에서 스코프(scope) 명시나 이슈 번호 Footer 연결은 GitHub Issues를 통한 이슈 트래킹을 병행하는 경우에만 실질적 가치를 발휘한다. 본 프로젝트는 단기 스프린트 특성상 GitHub Issues를 경량으로 운영하므로, 커밋 이력의 **가독성과 변경 의도의 명료한 분류**에 집중하여 Semantic Commit 규격을 아래와 같이 간소화하여 적용할 계획이다.

```
<type>: <subject>

[optional body]
[optional: closes #이슈번호  ← GitHub Issues 사용 시에만 작성]
```

* **`type`**: 필수. 아래 표에 정의된 타입 중 하나를 반드시 사용한다.
* **`scope`**: 선택 사항. 변경 범위가 특정 모듈에 한정되어 명시가 도움이 될 경우에만 `feat(lbs):` 형식으로 작성한다.
* **`body` / `footer`**: 선택 사항. 변경 이유, 관련 이슈 번호 등을 추가 서술할 때 작성한다.

| 타입 | 설명 | 예시 |
| :--- | :--- | :--- |
| `feat` | 새로운 기능 추가 | `feat: 헬스장 마커 탭 BottomSheet 슬라이딩 구현` |
| `fix` | 버그 수정 | `fix: 백그라운드 복원 시 타이머 오차 수정` |
| `refactor` | 기능 변경 없는 코드 리팩토링 | `refactor: FeedCard 컴포넌트 책임 분리` |
| `docs` | 문서 수정 | `docs: 유지보수 계획서 3.2절 브랜치 전략 수정` |
| `chore` | 빌드 시스템, 패키지 관리 등 | `chore: Vite 버전 업그레이드` |
| `hotfix` | 긴급 프로덕션 버그 수정 | `hotfix: JWT 만료 후 세션 오류 즉시 수정` |

> **[축소된 타입]** `style`과 `test` 타입은 별도 코드 포맷터(Prettier 자동화) 및 테스트 코드 작성 빈도가 낮은 1인 스프린트 환경의 특성을 감안하여 필수 타입 목록에서 제외한다. 필요 시 `chore` 또는 `refactor`로 대체하여 표현한다.

## 4. 변경관리 개요

### 4.1 변경 요청 절차

설계, 요구사항, 아키텍처 등에 대한 변경이 발생할 경우, 임의로 코드를 수정하는 것을 금지하고 아래의 변경 요청(Change Request, CR) 절차를 반드시 따를 계획이다.

#### 4.1.1 변경 요청 분류

| 구분 | 설명 |
| :--- | :--- |
| **Minor Change** | 로직 변경 없는 UI 수정, 문서 업데이트, 설정 파일 조정 등 |
| **Major Change** | 기능 요구사항(FR) 추가·삭제, DB 스키마 변경, API 인터페이스 변경, 아키텍처 수정 등 |

#### 4.1.2 변경 요청서 (CR) 작성 양식

Major Change에 해당하는 경우, GitHub Issues에 아래 양식을 사용하여 변경 요청서를 정식으로 등록할 계획이다.

```markdown
## CR 제목: [변경 요약]

## 변경 유형
- [ ] 기능 추가 (Feature Addition)
- [ ] 기능 수정 (Feature Modification)
- [ ] 기능 삭제 (Feature Removal)
- [ ] 아키텍처 변경 (Architecture Change)
- [ ] DB 스키마 변경 (Schema Change)

## 변경 요청 사유
[변경이 필요한 이유와 배경]

## 변경 대상 범위
[영향을 받는 컴포넌트, 테이블, API 목록]

## 예상 영향도
[다른 기능 또는 시스템 컴포넌트에 미치는 영향]

## 관련 요구사항 ID
[FR-XX-XX 또는 NFR-XX]
```

### 4.2 변경 사항 반영 및 추적 절차

#### 4.2.1 변경 사항 검토 및 스프린트 백로그 편입

1인 개발 애자일 스크럼 환경의 특성상 전통적인 CCB(변경 통제 위원회)를 별도로 운영하지 않는다. 대신, Major Change에 해당하는 변경 요청은 다음 스프린트 백로그(Sprint Backlog) 기획 단계에서 자체 검토를 거쳐 반영 여부를 결정한다. Trunk-based Development 전략에 따라 별도의 PR 생성 없이 `main` 브랜치에 직접 커밋하되, 아래의 자체 검증 절차를 따른다.

```
CR 등록 (GitHub Issues)
      ↓
영향도 분석 및 스프린트 백로그 편입 결정
      ↓
[진행 시] main 브랜치에서 직접 개발 착수 및 커밋
      ↓
기능 구현 후 자체 체크리스트(Self-Review) 검증
      ↓
Issues 닫기 + 필요 시 문서 베이스라인 갱신
```

  #### 4.2.2 추적성(Traceability) 확보 방침

  변경 사항의 시작점(CR)부터 코드 반영 시점까지 완전한 추적이 가능하도록 아래 규칙을 의무화할 계획이다.

  * **커밋 연결 (선택적 추적성)**: Major Change에 해당하는 핵심 구조 변경 시, 커밋 메시지 하단에 관련 이슈 번호를 기재하여 최소한의 추적성을 확보한다. (예: `closes #42`)
  * **릴리즈 노트**: 각 스프린트 릴리즈 시 `CHANGELOG.md`에 반영된 주요 변경 내역(Major Change 중심)을 기록하여 버전별 형상 이력을 유지한다.

### 4.3 변경 사항 관리 요령

#### 4.3.1 자가 검증(Self-Review) 및 CI 검증 체계

1인 Trunk-based Development 환경에서는 동료의 PR 코드 리뷰(Peer Review)를 수행할 수 없으므로, 코드를 `main` 브랜치에 커밋하기 전 아래의 자가 검증 체크리스트를 의무화하고 CI/CD 파이프라인의 자동화 빌드 검증으로 이를 보완한다.

```
자가 검증(Self-Review) 체크리스트:
  [ ] 기능이 요구사항 명세서(FR ID) 및 CR의 변경 요약과 일치하게 동작하는가?
       └ 코드 수정 시 FR ID 참조 기준:
          └ 지도/제보 관련 기능 → FR-01-01, FR-01-02
          └ 스톱워치/영속성 관련 기능 → FR-02-01, FR-02-02
          └ 대시보드 시각화 관련 기능 → FR-03-01, FR-03-02
          └ 소셜 피드/알림 관련 기능 → FR-04-01, FR-04-02
          └ 점포 관리자 기능 → FR-05-01
  [ ] 수정 코드가 기존 핵심 기능(타이머, 지도 등)에 사이드 이펙트를 주지 않는가?
  [ ] Supabase RLS 정책 변경을 수반하는 경우, 권한 없는 사용자의 접근이 차단되는지 로컬 테스트를 거쳤는가?
  [ ] 간소화된 Semantic Commit 컨벤션을 준수하여 커밋하였는가?
```

#### 4.3.2 베이스라인 (Baseline) 관리

베이스라인은 형상관리의 핵심 통제 기준점으로, 프로젝트 진행 중 아래 시점에 공식 베이스라인을 지정하고 GitHub Release 태그로 관리할 계획이다.

| :--- | :--- | :--- |

| **설계 베이스라인** | 아키텍처 및 세부 설계 완료 시 | `docs/design-v1.0` |

## 5. 배포 계획

### 5.1 배포 일정 및 방식

#### 5.1.1 배포 유형 및 기준

| 배포 유형 | 트리거 조건 | 배포 브랜치 | 대상 환경 |
| :--- | :--- | :--- | :--- |
| **정기 배포 (Sprint Release)** | 스프린트 종료 시점 기능 릴리즈 | `main` 직접 푸시 | Vercel (Production) |

#### 5.1.2 CI/CD 파이프라인 자동화 계획

**GitHub Actions**를 활용하여 아래의 자동화 파이프라인을 구축할 계획이다. 이를 통해 사람의 수동 개입을 최소화하고 배포의 일관성과 안정성을 확보한다.

```yaml
# CI/CD 파이프라인 흐름 (Trunk-based 1인 개발 환경)

[main 브랜치 직접 Push (또는 Commit) 시] → [GitHub Actions 자동 트리거]
  └── Lint 검사 및 타입 체크 (ESLint, TSC)
  └── 자체 검증 빌드 (vite build)
  └── 통과 시: Vercel 프로덕션 자동 배포 파이프라인 트리거

[배포 성공 후 자동화 (선택적 구현)]
  └── GitHub Release 자동 생성 및 태그 등록
  └── 배포 완료 알림
```

### 5.2 운영 환경관리 및 배포 전략

#### 5.2.1 환경 분리 방침

1인 풀스택 개발 환경에 최적화하여 복잡한 고정 Staging 환경은 생략하고, 아래 2개의 환경(Local, Production)으로 분리 운영한다. 각 환경은 독립된 Supabase 프로젝트 인스턴스와 환경 변수(`.env`)를 사용하며, Production 데이터베이스는 로컬 테스트 시 절대 접근하지 않는 것을 원칙으로 한다. 필요시 중간 점검을 위해 Vercel Preview Deploy를 활용한다.

| 환경 | 목적 | 브랜치 | Supabase 인스턴스 | 환경 특성 |
| :--- | :--- | :--- | :--- | :--- |
| **Local (Dev)** | 기능 개발 및 자체 테스트 | `main` (로컬) | Dev 인스턴스 | 개발자 개인 로컬 서버 가동 |
| **Preview (선택)** | 기능 병합 전 임시 통합 점검 | Vercel 임시 브랜치 | Dev 인스턴스 | Vercel에서 제공하는 임시 URL 활용 |
| **Production (Prod)** | 실 서비스 운영 및 정기 릴리즈 | `main` | Prod 인스턴스 | 전체 사용자 대상 정식 도메인 연동 |

#### 5.2.2 배포 안정성 확보 전략

배포 후 예상치 못한 장애가 발생할 경우를 대비하여 아래의 안정성 확보 방침을 수립한다.

* **롤백(Rollback) 전략**: Vercel 플랫폼의 강력한 즉시 롤백(Instant Rollback) 기능을 활용하여, 배포 후 Critical 장애가 감지될 경우 1분 이내엔 이전 정상 배포 버전으로 즉시 복원할 계획이다. **롤백 트리거 조건**: ❌ 전체 서비스 페이지 HTTP 5xx 오류율 5% 초과, ❌ 핵심 기능(스톱워치 시작, 지도 로딩, 소셜 피드) 중 1가지 이상이 지속적 완전 장애 상태 감지, ❌ P0 급 장애(락 및 Critical) Supabase 에러 로그 발생. 해당 조건 확인 후 Vercel 대시보드의 "이전 배포로 프로모트" 기능으로 즉시 복원한다.
* **배포 후 모니터링**: Production 배포 완료 후 최소 24시간 동안 서비스 오류율 및 핵심 기능(스톱워치, 지도 로딩, 소셜 피드) 정상 동작 여부를 집중 모니터링할 계획이다.
* **DB 마이그레이션 안전 처리**: Supabase DB 스키마 변경이 수반되는 배포는 반드시 로컬 개발 환경에서 마이그레이션 스크립트를 완벽히 검증한 후 진행하며, 되돌릴 수 없는 파괴적 마이그레이션(컬럼 삭제 등)은 배포 직전 Supabase 대시보드를 통한 수동 백업을 의무화할 계획이다.
