# 소프트웨어 요구분석 명세서 (Software Requirements Specification)

**프로젝트명**: LoGym (사용자 참여형 장소 기반 운동 관리 시스템)
**문서 기준**: IEEE Std 830 요구분석 표준 준수
**작성일자**: 2026-05-23

## 1. 프로젝트 개요

### 1.1 프로젝트명 및 목표
* **프로젝트명**: LoGym
* **목표**: 본 프로젝트는 기존 피트니스 센터의 기구 브랜드, 관리 상태 등 인프라 정보가 비대칭적으로 제공되는 한계를 해결하기 위해 기획되었다. 실제 헬스장의 인프라 정보(기구 맵)와 개인의 운동 기록(고정밀 타이머, 세트 데이터)을 유기적으로 결합하여, 사용자가 운동 장소를 탐색하는 비용을 줄이고 운동 세션을 안전하게 기록 및 분석할 수 있도록 지원한다. 더불어 소셜 피드를 통해 사용자 간의 지지를 이끌어내는 피트니스 소셜 플랫폼을 지향한다.

### 1.2 프로젝트 주요 기능
1. **Kakao Map SDK 기반 위치 탐색**: 위치 기반 주변 헬스장 탐색 및 기구 마스터 정보 오버레이 시각화.
2. **사용자 참여형 정보 제보 시스템**: 집단지성을 이용한 헬스장 보유 기구 등록 및 고장 상태 신고/수정 기능.
3. **실시간 인프라 데이터 동기화**: Supabase Realtime Listener를 통한 별도 새로고침 없는 실시간 인프라 현황 반영.
4. **Split-Ring 게이지 및 SVG 근육 맵 시각화**: 당일 운동 진척도를 직관적으로 시각화하는 대시보드와 경량화된 SVG 기반 부위별 자극 히트맵 제공.
5. **고정밀 스톱워치 및 백그라운드 복원 기록**: 1/100초 단위 기록 엔진과 LocalStorage 백업을 통한 앱 강제 종료 시의 데이터 무손실 복구 메커니즘.
6. **RLS 보안 기반 소셜 피드**: 팔로우 기반 '오운완' 게시물 공유 및 실시간 상호작용(좋아요, 댓글).

## 2. 프로젝트 요구사항

### 2.1 요구사항 도출 (Requirements Elicitation)
본 요구사항은 시장 실태 분석 및 타겟 사용자 그룹 인터뷰(FGI) 과정을 통해 도출되었으며, 사용자가 개발자에게 실제로 제시한 핵심 니즈(Pain Point)를 시스템 요구사항으로 매핑하여 정리하였다.

* **요구사항 1: 장소 위치정보 제공**
  * **사용자 요구사항**: "주변 헬스장들의 위치를 알고 싶어요."
  * **시스템 요구사항**: 지도 API 기반 위치정보와 운동기관 코드 장소들 목록화 제공 및 검색 기능 제공.

* **요구사항 2: 헬스장 인프라의 투명한 확인**
  * **사용자 요구사항**: "헬스장에 방문하기 전에 내가 오늘 하려는 운동 기구(예: 파워 렉, 특정 브랜드 머신)가 충분히 있는지, 고장 나지는 않았는지 미리 알고 싶어요."
  * **시스템 요구사항**: 위치 기반으로 주변 헬스장의 위치를 파악하고 선택한 헬스장의 기구 인프라 현황을 확인할 수 있으며, 사용자 집단지성을 이용해 기구의 추가 및 고장 상태를 실시간으로 제보하고 동기화하는 시스템 제공.

* **요구사항 3: 운동 기록의 중단 없는 보존**
  * **사용자 요구사항**: "운동 중에 스톱워치를 켜놓고 세트를 기록하다가, 잠깐 전화를 받거나 다른 앱을 켜면 앱이 꺼지면서 기록이 초기화되는 게 너무 불편해요."
  * **시스템 요구사항**: 모바일 OS에 의해 앱이 강제 종료되거나 백그라운드로 전환되더라도 타이머 오차 없이 즉각 복구할 수 있는 LocalStorage 백업 기반의 고정밀 스톱워치 기록 엔진 제공.

* **요구사항 4: 직관적인 운동 진척도 확인**
  * **사용자 요구사항**: "오늘 목표한 운동량을 얼마나 달성했는지, 내 몸의 어떤 부위를 가장 많이 자극했는지 복잡한 수치 대신 한눈에 알아보기 쉽게 확인하고 싶어요."
  * **시스템 요구사항**: 당일 운동 진척도를 원형 게이지(Split-Ring) 형태로 제공하고, 누적 운동 볼륨에 따라 2D 인체 벡터(SVG) 이미지의 부위별 색상을 다르게 렌더링하는 시각화 대시보드 제공.

* **요구사항 5: 운동 지속성을 위한 소셜 기능**
  * **사용자 요구사항**: "매일 혼자 헬스장에 가려니 동기부여가 안 돼요. 내 운동 완료 기록을 친구들과 공유하고 서로 응원하면서 자극을 받고 싶어요."
  * **시스템 요구사항**: 사용자 간 팔로우 관계를 바탕으로 '오운완(오늘 운동 완료)' 인증 사진과 세션 로그를 공유하고 인터랙션(좋아요, 댓글)을 주고받을 수 있는 RLS 보안 정책 기반 소셜 피드 제공.

### 2.2 요구사항 분석 (Requirements Analysis)
이번 LoGym 프로젝트에서는 사용자와 시스템 측면에서의 요구사항을 다음과 같이 분석하였습니다.

먼저 **사용자 요구사항**은 실질적인 사용자가 LoGym을 통해 기대하는 기능과 경험을 중심으로 도출되었습니다. 예를 들어, 사용자는 헬스장에 방문하기 전 주변 헬스장의 위치와 해당 헬스장이 보유한 기구 라인업 및 고장 여부를 미리 파악하여 방문 계획을 세우길 원합니다. 또한 운동 세션(세트, 무게, 시간)의 진행 결과를 수치 나열이 아닌 원형 게이지·근육 히트맵 등 시각화된 형태로 직관적으로 확인하고자 합니다. 나아가, 운동이 단순한 개인 활동을 넘어 '오운완(오늘 운동 완료)' 인증 사진과 세션 기록을 친구들과 공유하고 서로 응원을 주고받을 수 있는 커뮤니티 기능을 통해 지속적인 운동 동기를 유지하고자 하는 니즈도 확인되었습니다.

이러한 사용자 요구를 바탕으로 정의된 **시스템 요구사항**은 기능 구현의 기술적 방향을 결정짓는 기준이 됩니다. 우선, 지도 API를 활용하여 현재 GPS 위치 기반으로 주변 헬스장을 지도에 마커로 표시하고, 선택한 헬스장의 기구 현황을 조회·제보할 수 있는 LBS 기능이 필요합니다. 또한 사용자가 앱을 백그라운드로 전환하거나 강제 종료하더라도 운동 기록이 손실되지 않도록, `requestAnimationFrame` 기반 고정밀 스톱워치와 LocalStorage 이중 백업 로직을 제공해야 합니다. 수집된 운동 데이터는 Split-Ring 게이지·SVG 인체 히트맵 등으로 시각화하여 사용자가 운동 효과를 명확히 인지할 수 있도록 해야 하며, 소셜 피드 기능은 Supabase Row Level Security를 통해 사용자 인증 기반의 권한 통제를 적용하여 데이터 보안을 보장해야 합니다.

위와 같은 분석 결과를 바탕으로 다음 장 **2.3 기능적 요구사항**에서 구체적인 기능을 정의합니다.

### 2.3 기능적 요구사항 (Functional Requirements)

* **전제조건**: 기기의 GPS 위치 권한이 허용되어 있고, 사용자는 로그인 상태여야 함.
* **기본 흐름 (Basic Flow)**:
  1. 액터가 지도 탭에 진입하면, 시스템은 액터의 현재 GPS 좌표를 수신한다.
  2. 시스템은 수신된 좌표 기준 반경 3km 이내의 등록된 헬스장 목록을 DB에서 조회한다.
  3. 시스템은 지도 화면 위에 헬스장들의 위치를 커스텀 마커로 표시하고, 화면 하단의 BottomSheet 목록에 주변 헬스장 명단을 렌더링한다.
  4. 액터가 특정 헬스장 마커를 터치하거나 하단 리스트의 특정 헬스장 항목을 선택한다.
  5. 시스템은 BottomSheet를 해당 헬스장의 요약 정보 컴포넌트(상호명, 간단한 혼잡도 등)로 업데이트하여 출력한다.
  6. 액터가 요약 카드를 터치하여 해당 헬스장의 독립적인 '상세 정보 페이지'로 화면을 이동한다.
* **대체/예외 흐름 (Alternative/Exception Flow)**:
  * **(위치 권한 거부 시)**: 시스템은 서울 강남역 등 디폴트 좌표를 기준으로 지도를 렌더링하고, 상단에 '위치 기반 서비스 활성화 권고' 모달을 띄운다.
  * **(반경 내 등록 헬스장 부재 시)**: 시스템은 지도 상에 빈 화면 상태(Empty State) 가이드를 노출하고 사용자가 검색 창을 통해 수동으로 헬스장을 검색하거나 지도를 드래그하여 탐색하도록 대기한다.

### 4.2 주요기능-2: 헬스장 상세 정보 조회 및 기구 상태 제보 시나리오
* **액터**: 일반 사용자 (회원)
* **전제조건**: 액터가 헬스장 탐색을 거쳐 특정 헬스장의 상세 정보 페이지에 진입한 상태임.
* **기본 흐름 (Basic Flow)**:
  1. 시스템은 해당 헬스장의 고유 ID를 기반으로 등록된 전체 운동 기구 인프라 목록과 최근 24시간 실시간 제보 현황을 DB에서 조회하여 상세 정보 뷰에 렌더링한다.
  2. 액터가 목록을 조회하던 중 실제 기구의 상태(고장/정상)와 화면상의 표기가 일치하지 않음을 식별하고, 해당 기구 옆의 '상태 제보' 버튼을 클릭한다.
  3. 액터가 기구 상태값(고장/정상)을 선택 및 토글하고 '제보 완료'를 제출한다.
  4. 시스템은 DB의 `gym_equipments_reports` 테이블에 제보 데이터를 Insert하고 실시간 상태를 갱신한다.
  5. 백엔드는 Realtime 채널을 통해 이 변경 이벤트를 동일 헬스장 상세 페이지를 조회 중인 타 사용자들의 화면에 즉시 반영(State Update)하고, 동시에 해당 점포 관리자(owner)의 알림 서랍으로 제보 접수 알림을 푸시한다.
* **대체/예외 흐름 (Alternative/Exception Flow)**:
  * **(중복 제보 제어)**: 24시간 이내에 동일한 액터가 동일한 헬스장의 동일 기구에 대하여 동일한 상태값을 제보하려는 경우, 시스템은 "이미 접수된 상태 정보입니다" 안내 Alert를 표출하고 DB Insert 트랜잭션을 전송하지 않은 채 종료한다.

### 4.3 주요기능-3: 스톱워치 가동 중 강제 종료 복원 시나리오 (데이터 영속성)
* **액터**: 일반 사용자, 모바일 운영체제 (System Exception)
* **전제조건**: 액터가 스톱워치를 가동하고 벤치프레스 2세트를 마친 상태임.
* **기본 흐름 (Basic Flow)**:
  1. 전화 수신 등 외부 요인으로 브라우저가 백그라운드로 전환되고 OOM에 의해 강제 킬(Kill) 됨.
  2. 사용자가 10분 뒤 앱을 재실행하여 Session View에 진입함.
  3. 시스템은 React 컴포넌트 마운트 단계에서 `LocalStorage`를 파싱함.
  4. 시스템은 파싱한 `lastStartTime`과 현재의 `Date.now()`를 계산하여 타이머 UI에 오차 없는 경과 시간을 즉시 렌더링함.
  5. 저장된 세트 데이터를 불러와 사용자가 끊김 없이 3세트 기록을 이어나가도록 UI를 복원함.
* **대체/예외 흐름**:
  * **(LocalStorage 손상 시)**: JSON 파싱 에러 발생 시, 비정상적 상태 데이터는 무시(`removeItem`)하고 새로운 빈 세션을 초기화한다.

### 4.4 주요기능-4: 소셜 피드 작성 및 RLS 보안 검증 시나리오
* **액터**: 일반 사용자, 악의적 사용자(Attacker)
* **전제조건**: 사용자는 운동 세션을 완료하고 '오운완 인증' 피드를 작성하고자 함.
* **기본 흐름 (Basic Flow)**:
  1. 액터가 인증 사진(Cloudinary 업로드)과 피드 본문을 입력 후 제출한다.
  2. 프론트엔드는 JWT 토큰이 헤더에 포함된 상태로 Supabase POST 요청을 보낸다.
  3. Postgres RLS 정책은 헤더의 `auth.uid()`를 추출하여 피드 row의 `user_id` 필드와 일치하는지 검증 후 Insert를 승인한다.
* **대체/예외 흐름**:
  * **(악의적 조작 접근 시)**: 액터(Attacker)가 클라이언트 코드를 변조하여 타인의 `user_id`를 삽입해 Insert를 시도하는 경우, Supabase RLS의 `WITH CHECK (auth.uid() = user_id)` 규칙이 위반되므로 DB 엔진 레벨에서 `403 Forbidden` 트랜잭션 에러를 반환하고 즉시 거부한다.

## 3. UI/UX 설계 방향

### 3.2 UX 설계 방향 (User Experience)
* **입력 플로우 간소화**: 운동 세션 기록 시 불필요한 텍스트 타이핑을 최소화하기 위해, 무게 및 횟수 입력 등은 직관적인 증감 버튼이나 숫자 패드 기반의 간소화된 플로우로 구성하여 운동의 흐름이 끊기지 않도록 유도한다.
* **원클릭 피드 연동 및 편의성 증진**: 오운완(오늘 운동 완료) 피드 작성 시, 사용자가 직접 운동 내역을 입력할 필요 없이 당일 완료된 운동 세션 기록을 원클릭으로 호출하여 자동으로 피드 본문에 바인딩할 수 있는 연동 플로우를 설계해 사용자 편의성을 극대화할 계획이다.
* **위치 기반 직관적 지도 탐색**: 사용자가 복잡한 검색이나 화면 전환 없이 직관적으로 주변 헬스장을 인지할 수 있도록, Kakao Map SDK 상에서 사용자의 현재 위치 반경 내 헬스장들을 커스텀 마커로 오버레이한다. 특정 마커를 탭할 시 화면 전체를 바꾸는 대신 지도 하단에서 헬스장 핵심 요약 정보가 담긴 슬라이드업 카드(BottomSheet)가 부드러운 슬라이딩 트랜지션과 함께 올라오도록 설계하여 탐색 피로도를 최소화할 계획이다.
* **데이터 신뢰성을 위한 중복 제보 방지 로직**: 사용자가 헬스장이나 운동 기구 상태를 제보할 때, 동일한 위치나 기구에 대해 이미 다른 사용자가 등록한 제보 내역이 있다면 지도 상에서 이를 실시간으로 감지하고 신규 등록 대신 기존 제보에 대한 '동의/상태 변경' 처리를 유도함으로써 데이터 중복 축적을 방지하고 커뮤니티 정합성을 유지하도록 설계한다.
* **마이크로 인터랙션**: 고정밀 스톱워치 타이머의 [시작/일시정지] 버튼 조작 시, 즉각적인 시각적 스케일 다운(0.95) 모션 애니메이션과 타이머 상태별 테두리 색상 전환(예: 진행 중일 때는 활성화 블루, 일시정지일 때는 소프트 그레이)을 적용하여, 웹/PWA 환경에서도 사용자에게 명확하고 즉각적인 조작 피드백을 체감하도록 설계한다.

## 4. 프로젝트 기능 시나리오 (Use Case Scenarios)

### 4.1 주요기능-1: 위치 기반 주변 헬스장 탐색 및 선택 시나리오
* **액터**: 일반 사용자 (회원)
* **전제조건**: 기기의 GPS 위치 권한이 허용되어 있고, 사용자는 로그인 상태여야 함.
* **기본 흐름 (Basic Flow)**:
  1. 액터가 지도 탭에 진입하면, 시스템은 액터의 현재 GPS 좌표를 수신한다.
  2. 시스템은 수신된 좌표 기준 반경 3km 이내의 등록된 헬스장 목록을 DB에서 조회한다.
  3. 시스템은 지도 화면 위에 헬스장들의 위치를 커스텀 마커로 표시하고, 화면 하단의 BottomSheet 목록에 주변 헬스장 명단을 렌더링한다.
  4. 액터가 특정 헬스장 마커를 터치하거나 하단 리스트의 특정 헬스장 항목을 선택한다.
  5. 시스템은 BottomSheet를 해당 헬스장의 요약 정보 컴포넌트(상호명, 간단한 혼잡도 등)로 업데이트하여 출력한다.
  6. 액터가 요약 카드를 터치하여 해당 헬스장의 독립적인 '상세 정보 페이지'로 화면을 이동한다.
* **대체/예외 흐름 (Alternative/Exception Flow)**:
  * **(위치 권한 거부 시)**: 시스템은 서울 강남역 등 디폴트 좌표를 기준으로 지도를 렌더링하고, 상단에 '위치 기반 서비스 활성화 권고' 모달을 띄운다.
  * **(반경 내 등록 헬스장 부재 시)**: 시스템은 지도 상에 빈 화면 상태(Empty State) 가이드를 노출하고 사용자가 검색 창을 통해 수동으로 헬스장을 검색하거나 지도를 드래그하여 탐색하도록 대기한다.

### 4.2 주요기능-2: 헬스장 상세 정보 조회 및 기구 상태 제보 시나리오
* **액터**: 일반 사용자 (회원)
* **전제조건**: 액터가 헬스장 탐색을 거쳐 특정 헬스장의 상세 정보 페이지에 진입한 상태임.
* **기본 흐름 (Basic Flow)**:
  1. 시스템은 해당 헬스장의 고유 ID를 기반으로 등록된 전체 운동 기구 인프라 목록과 최근 24시간 실시간 제보 현황을 DB에서 조회하여 상세 정보 뷰에 렌더링한다.
  2. 액터가 목록을 조회하던 중 실제 기구의 상태(고장/정상)와 화면상의 표기가 일치하지 않음을 식별하고, 해당 기구 옆의 '상태 제보' 버튼을 클릭한다.
  3. 액터가 기구 상태값(고장/정상)을 선택 및 토글하고 '제보 완료'를 제출한다.
  4. 시스템은 DB의 `gym_equipments_reports` 테이블에 제보 데이터를 Insert하고 실시간 상태를 갱신한다.
  5. 백엔드는 Realtime 채널을 통해 이 변경 이벤트를 동일 헬스장 상세 페이지를 조회 중인 타 사용자들의 화면에 즉시 반영(State Update)하고, 동시에 해당 점포 관리자(owner)의 알림 서랍으로 제보 접수 알림을 푸시한다. **(관련 요구사항 검증: FR-01-02, FR-04-02, FR-05-01)**
* **대체/예외 흐름 (Alternative/Exception Flow)**:
  * **(중복 제보 제어)**: 24시간 이내에 동일한 액터가 동일한 헬스장의 동일 기구에 대하여 동일한 상태값을 제보하려는 경우, 시스템은 "이미 접수된 상태 정보입니다" 안내 Alert를 표출하고 DB Insert 트랜잭션을 전송하지 않은 채 종료한다.

### 4.3 주요기능-3: 스톱워치 가동 중 강제 종료 복원 시나리오 (데이터 영속성)
* **액터**: 일반 사용자, 모바일 운영체제 (System Exception)
* **전제조건**: 액터가 스톱워치를 가동하고 벤치프레스 2세트를 마친 상태임.
* **기본 흐름 (Basic Flow)**:
  1. 전화 수신 등 외부 요인으로 브라우저가 백그라운드로 전환되고 OOM에 의해 강제 킬(Kill) 됨.
  2. 사용자가 10분 뒤 앱을 재실행하여 Session View에 진입함.
  3. 시스템은 React 컴포넌트 마운트 단계에서 `LocalStorage`를 파싱함.
  4. 시스템은 파싱한 `lastStartTime`과 현재의 `Date.now()`를 계산하여 타이머 UI에 오차 없는 경과 시간을 즉시 렌더링함.
  5. 저장된 세트 데이터를 불러와 사용자가 끊김 없이 3세트 기록을 이어나가도록 UI를 복원함.
* **대체/예외 흐름**:
  * **(LocalStorage 손상 시)**: JSON 파싱 에러 발생 시, 비정상적 상태 데이터는 무시(`removeItem`)하고 새로운 빈 세션을 초기화한다.

### 4.4 주요기능-4: 소셜 피드 작성 및 RLS 보안 검증 시나리오
* **액터**: 일반 사용자, 악의적 사용자(Attacker)
* **전제조건**: 사용자는 운동 세션을 완료하고 '오운완 인증' 피드를 작성하고자 함.
* **기본 흐름 (Basic Flow)**:
  1. 액터가 인증 사진(Cloudinary 업로드)과 피드 본문을 입력 후 제출한다.
  2. 프론트엔드는 JWT 토큰이 헤더에 포함된 상태로 Supabase POST 요청을 보낸다.
  3. Postgres RLS 정책은 헤더의 `auth.uid()`를 추출하여 피드 row의 `user_id` 필드와 일치하는지 검증 후 Insert를 승인한다.
* **대체/예외 흐름**:
  * **(악의적 조작 접근 시)**: 액터(Attacker)가 클라이언트 코드를 변조하여 타인의 `user_id`를 삽입해 Insert를 시도하는 경우, Supabase RLS의 `WITH CHECK (auth.uid() = user_id)` 규칙이 위반되므로 DB 엔진 레벨에서 `403 Forbidden` 트랜잭션 에러를 반환하고 즉시 거부한다.

## 5. 기능별 세부 설계 (Architecture & Design)

### 5.1 기능-1 세부 설계: LBS 지도 및 실시간 동기화
* **기능 설명**: 사용자의 현재 위치를 기반으로 주변 헬스장의 기구 인프라 정보를 지도에 시각화하고, 데이터 변동 시 실시간으로 동기화하여 제공하는 기능입니다.
* **기능 구성 요소**:
  1. 지도 렌더링: GPS 데이터 및 지도 Web SDK를 활용한다.
  2. 실시간 동기화: 백엔드의 Realtime 채널을 활용하여 데이터 변경 이벤트를 수신한다.
* **기능 상세 설계**:
  5.1.1 지도 컴포넌트 마운트
  앱은 사용자 위치 좌표를 실시간으로 추적하여 Kakao Map 컨테이너에 센터 좌표를 바인딩함.
  주변 반경 내 헬스장 마커를 렌더링하고 커스텀 오버레이를 통해 기구 요약 정보를 제공하도록 설계.
  5.1.2 실시간 데이터 브로드캐스팅
  기구 고장 신고 등의 이벤트 발생 시, 클라이언트는 `gym_equipments` 테이블의 INSERT/UPDATE 이벤트를 실시간으로 구독하여 사용자의 새로고침 조작 없이 즉각적으로 화면의 상태(State)를 갱신하도록 설계.

### 5.2 기능-2 세부 설계: 스톱워치 루프 및 복원 알고리즘
* **기능 설명**: 모바일 환경에서 정밀한 운동 세션 시간을 측정하며, 예기치 않은 앱 강제 종료 시에도 데이터를 안전하게 복구하여 연속적인 측정을 보장하는 기능입니다.
* **기능 구성 요소**:
  1. 타이머 엔진: 브라우저의 렌더링 프레임 최적화 API를 활용하여 정밀도를 확보한다.
  2. 상태 직렬화: 세션 데이터를 JSON 포맷으로 가공하여 기기의 로컬 저장소에 캐싱한다.
* **기능 상세 설계**:
  5.2.1 스톱워치 루프 엔진
  일반적인 타이머 API 사용 시 발생하는 모바일 백그라운드 스로틀링(Throttling) 현상을 방지하기 위해, `requestAnimationFrame(rAF)`을 활용한 커스텀 로직을 설계하여 1/100초 단위의 정밀도를 보장함.
  5.2.2 세션 데이터 영속성 (LocalStorage)
  운동 세션의 절대 시작 시간(`lastStartTime`) 및 세트별 누적 데이터를 매 변경 시점마다 직렬화하여 기기의 LocalStorage에 동기화함. 앱 재가동 시 이를 파싱하여 타이머 오차를 보정하고 중단 직전의 상태로 완벽히 복원 가능하도록 설계.

### 5.3 기능-3 세부 설계: Split-Ring 및 SVG 히트맵 시각화
* **기능 설명**: 사용자의 운동 진척도와 부위별 자극량을 복잡한 수치가 아닌 직관적인 그래픽 인터페이스로 시각화하여 제공하는 대시보드 기능입니다.
* **기능 구성 요소**:
  1. 진척도 게이지: 원형 벡터(SVG) 요소를 활용하여 목표 달성률을 시각적으로 렌더링한다.
  2. 근육 히트맵: 2D 인체 벡터 이미지를 활용하여 자극 부위를 색상 매핑으로 표현한다.
* **기능 상세 설계**:
  5.3.1 Split-Ring 애니메이션 설계
  `<circle>` 태그의 `stroke-dasharray`를 활용하여 원의 둘레를 고정치로 정의하고, 사용자의 목표 대비 현재 완료된 세트의 백분율에 비례하여 `stroke-dashoffset` 속성을 제어함으로써 동적인 게이지 바를 렌더링하도록 설계.
  5.3.2 2D SVG 데이터 바인딩
  DB에서 집계된 사용자의 부위별 누적 볼륨 데이터를 프론트엔드의 사상표(Dictionary)와 매칭함. 자극 임계값 비율에 따라 각 근육 부위의 `<path>` 태그 `fill` 색상(예: 고자극 빨간색, 중자극 주황색, 저자극 회색 등)을 동적으로 치환하도록 설계.

### 5.4 기능-4 세부 설계: 논리 스키마 및 RLS 보안
* **기능 설명**: 관계형 데이터베이스 구조를 기반으로 데이터의 참조 무결성을 유지하며, 데이터베이스 엔진 레벨의 접근 통제(RLS)를 통해 개인정보와 소셜 피드의 보안을 보장하는 아키텍처입니다.
* **기능 구성 요소**:
  1. 관계형 스키마 설계: 유저, 헬스장, 기구 인프라, 운동 일지 간의 관계를 정규화하여 정의한다.
  2. 보안 정책(RLS): 데이터베이스 엔진에서 직접 사용자 권한별 쿼리 접근을 통제한다.
* **기능 상세 설계**:
  5.4.1 ERD 중심 관계 설정
  1:N 및 1:1 관계를 바탕으로 테이블 구조를 설계함. `users`와 `workout_logs`, `gyms`와 `gym_equipments` 등의 참조 무결성을 강제하도록 논리 스키마 구축.
  5.4.2 RLS(Row Level Security) 접근 통제
  전체 테이블에 RLS를 활성화하여, 데이터 조작(INSERT/UPDATE) 트랜잭션 요청 시 JWT 토큰의 `auth.uid()`와 레코드 소유자의 `user_id` 일치 여부를 검증하도록 `WITH CHECK` 규칙을 설계. 소셜 피드의 경우, 팔로우 관계(`follows` 테이블)에 기반한 동적 조회(`USING`) 보안 정책을 적용하여 악의적인 접근을 원천 차단함.
